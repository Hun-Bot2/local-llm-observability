# Architecture V1 + V2

Last updated: 2026-04-09

## Core Context

This project does **not** start from a website-first workflow.

The real source of truth is:

1. my original Korean blog posts are in **another repository / local folder**
2. the translation system must read from that local source
3. translated files must be written back to my local blog folder so I can review them
4. in V1, I will **commit manually**
5. in V2, after I finish writing and commit a post, the system should detect that and translate automatically

---

## V1

## Goal

Translate all existing Korean blog posts first, calculate translation quality by comparing them with the original Korean markdown files, write the translated files back into my local folder, and notify me about the pipeline state so I can review and commit manually.

## V1 Requirements

### Source

- Original Korean blog posts live in another repository on my local machine.
- The translator must scan that local folder directly.

### Translation

- Translate all target blog posts.
- Preserve MDX / Markdown structure.
- Preserve frontmatter structure.
- Translate code comments carefully without breaking code.

### Quality Checking

- Compare translated files against the original Korean markdown files.
- Calculate translation quality after translation is complete.
- Quality should include:
  - structure preservation
  - glossary correctness
  - length sanity
  - semantic similarity
  - code/comment handling quality

### Output

- Save translated blog files into my local blog folder.
- I must be able to open them, inspect them, and decide whether to commit.

### Workflow Ownership

- V1 is **manual commit flow**
- translation is automatic
- final git commit is done by me

### Notifications / Alarm

I need an alarm / notification system that tells me:

- translation started
- translation is currently running
- translation finished successfully
- translation failed
- quality score is low
- specific files have problems

### AI Agent Role

- I want the notification / monitoring / checking flow to behave like an AI agent
- candidate model family: `Gemma 4`

### Recommendation for V1 Agent Design

Use AI only where it adds value:

- translation
- optional translation critique
- optional explanation of why quality is low

Do **not** use AI as the only pipeline controller for:

- file scanning
- state transitions
- writing output files
- commit logic
- retry logic

Those should remain deterministic Python logic.

## V1 Runtime Flow

1. Scan the original local blog repository.
2. Find all Korean source MDX files.
3. Translate all files.
4. Run quality checking against the original Korean files.
5. Save translated files into the local blog folder.
6. Send status notifications:
   - started
   - running
   - completed
   - failed
   - low quality warning
7. I manually review the translated files.
8. I manually commit if they are acceptable.

## V1 Architecture

- Source repo: another local blog repository
- Translator pipeline: Python
- Translation model:
  - EN candidate: `Gemma 4`
  - JP candidate: `Qwen 3`
- Quality checker: Python scoring pipeline
- Notification system: Slack webhook or similar
- Output location: local blog folder
- Final commit: manual by me

## V1 Deliverables

- local source scanner
- batch translator
- quality scoring report
- notification system
- local output writer
- manual review workflow

## V2

## Goal

After the full existing blog archive is translated, I will continue writing 2 to 4 blog posts per week. When I finish writing a post and commit it, the AI agent should detect the change and translate it automatically.

## V2 Requirements

### Trigger

- I write a Korean blog post.
- I commit the post.
- The system detects that new or changed Korean content exists.

### Automation

- Automatically detect newly added or modified Korean blog posts.
- Automatically run translation for those changed files.
- Automatically calculate quality.
- Automatically notify me about result status.

### Intelligent Detection

The agent should catch:

- new blog posts
- updated blog posts
- changed sections or paragraphs if possible
- missing translations
- stale translations after source update

### Quality Gate

Before finalizing translation results, the system should:

- calculate quality
- detect suspicious outputs
- warn on low-quality translations
- make it easy for me to review the results

### Output Behavior

After automatic translation:

- translated files should be saved into the blog repository
- I should still be able to review them
- commit behavior can remain semi-manual at first if needed

## V2 Runtime Flow

1. I finish writing a Korean blog post.
2. I commit it.
3. A watcher or trigger detects changed source content.
4. The pipeline finds new or changed Korean MDX files.
5. Only changed files or changed sections are translated.
6. Quality is calculated automatically.
7. Notifications are sent:
   - started
   - completed
   - failed
   - low quality warning
8. Translated files are written into the blog repository.
9. I review and finalize the result.

## V2 Trigger Options

### Stronger Portfolio Option

- GitHub push or webhook based trigger
- Cloud Run controller
- RunPod worker
- Postgres tracking and cache
- Slack notifications

### Simpler Local Option

- local file watcher
- local Python controller
- RunPod worker
- local or managed Postgres
- Slack notifications

For portfolio strength, the cloud-triggered option is better.

## V2 Architecture

- Trigger:
  - GitHub webhook or GitHub Actions
  - or local watcher for an earlier version
- Controller:
  - Python service
- GPU translation:
  - RunPod
- Cache / metrics:
  - Postgres
- Notification:
  - Slack webhook
- Review:
  - human approval still allowed

## V2 Deliverables

- changed-file detector
- incremental translation runner
- auto-quality check
- notification integration
- automatic translation after commit

---

## Recommended Responsibility Split

### Deterministic Python Logic

Use standard code for:

- scanning source files
- diff detection
- cache lookup
- saving translated files
- quality score calculation
- pipeline state tracking
- notifications
- commit / PR / review flow

### AI / LLM Logic

Use models for:

- translation
- optional translation review
- optional rewrite suggestion
- optional explanation of quality problems

---

## Best Practical Build Order

### Phase 1

- build local source scanner
- build full archive translation flow
- build quality checker
- build notification system
- keep commit manual

### Phase 2

- build changed-file detection
- build automatic trigger after commit
- build incremental translation
- add quality gate and alerting
- optionally add auto-commit or PR flow later

---

## Summary

### V1

- translate all existing local Korean blog posts
- check quality against original Korean markdown
- write translated files into local folder
- notify me about running / finished / failed / low quality
- I review and commit manually

### V2

- after I write and commit new blog posts
- the AI agent detects changed content automatically
- translation runs automatically
- quality is checked automatically
- notifications are sent automatically

This is the correct roadmap because it starts from a controlled manual workflow, then moves into automation only after the translation pipeline is reliable.
