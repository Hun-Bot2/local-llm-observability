さらに,個人Study Repoに付属していた Slack Botを分離してRAG Repoに統合しました.Mono Repoとして構築しようとしましたが,その後拡張性を考慮して,別々にRepoを作成して管理することにしました.

FAISSをどこで使ったのか? ->Leetcodeの問題を受け,埋め込みベクトルに変換し,FAISSインデックスに保存し,Github Repoにパッケージとしてアップロードしました.

まず,私が収集した #Leetcode問題は約3000件 (無料問題のみ) であり,問題を保存した形式はjsonlファイルであった.当然のことですが, LLMがこの問題をすべてスクロールして類似した問題を検索することは非効率です. "検索"するには,ベクトリー化して保存するスペースが必要であり,ローカルベクトルDBを探しました.

**ChromaDB**:SQLiteをエンジンとして使用するオープンソースのベクトルストレージです.Pythonのネイティブ環境では使用が非常に簡単で,メタデータフィルタリング機能が強力です.

**FAISS** (Facebook AI類似性検索): メタはFacebookで開発された高性能類似性検索ライブラリです. DBRAよりもアルゴリズムライブラリに近いため,メモリ内の検索速度は圧倒的です.

1.Langchainを勉強中に使ってみたかったので使いました. (ブログ3ページ参照してください.) 2.Github Actionsに適したベクトルDBが必要になり,FAISSは単純なライブラリ形式でのインデックスファイルのみを管理できていたので, 3.構築された問題は,追加頻度が低い時に構築された場合,Reedmanを模擬的に使っているため,高速 FAISSを選択しました.

問題:GitHub Actions自身にはファイル保存機能があるが,なぜAWS S3を使用したのでしょうか?

1. **データ継続性:** GitHub Artifacts は一定の期間後削除されますが,S3 は永久保存可能です. 2. **作業間解明:**Heavy Job と Light Job の実行環境を物理的に隔離し,システム柔軟性を確保しました. S3 を中間媒体として利用し,データ転送と受信プロセスの結合を低減させることで,各作業が相互に依存せず,独立した実行可能な構造が必要でした. 3. **外向拡張性:**後にスラック以外のWeb大板などを作るとき,S3 のデータを直接APIに呼び出す拡張性を考慮しました.

##重労働・軽労働の解消 - **Heavy Job:** Leetcode問題を収集し,埋め込みベクトルを作成した後, FAISS インデックスを構築します.この作業は時間がかかるし,頻繁に実行されません. - **Light Job:** FAISS インデックスを呼び出し,類似した問題を検索し,類似した問題と llmで生成した理由をスラックに転送します.この作業は頻繁に実行され,迅速な対応が必要です.
