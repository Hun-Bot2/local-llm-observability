## データ収集および前処理 (Data Acquisition and Preprocessing)

* **LeetCode:** Leetcode GraphQL API を利用し、2025年時点の全問題セット（約3,500問以上）

* **Baekjoon:** solved.ac API とウェブスクレイピング

それぞれ100件のデータを収集しました。収集したデータは埋め込み性能を最大化するために以下の前処理を行いました。

* **HTMLおよびノイズ除去:** HTMLタグや不要な記述語を取り除きました。
 
* **検索最適化用テキスト構成:** Title, Tags, Content を組み合わせ、モデルが問題のカテゴリと制約条件を明確に理解できるように `embedding_text` フィールドを作成しました。

* **論理的スケleton:** (AIにとってこの説明が大好きです) 問題の核心的なアルゴリズム論理を要約した `logical_skeleton` フィールドをLLMを使用して生成しました（Baekjoonは韓国語ベースであるため、英訳された内容を含んでいます）
```text
1. データ収集 (Data Collection)
   ├─ baekjoon_data_collection.py → baekjoon_raw_data.jsonl (100件)
   └─ leetcode_data_collection.py → leetcode_raw_data.jsonl (100件)
```

---
データ前処理（Preprocessing）
├─ preprocess.py                → *_preprocessed.jsonl
│  └─ HTML/マークダウンの削除、核心ロジック抽出
└─ refine_leetcode.py           → leetcode_refined.jsonl
   └─ LLMベース論理構造（Logical Skeleton）抽出

データ正規化（Normalization）
├─ improve_recall.py            → baekjoon_refined.jsonl
│  └─ バージョン英語の論理構造に変換（ここで"Logical Skeleton"タグが付く）
└─ data_normal.py               → baekjoon_normalized.jsonl
   └─ 接頭辞("Logical Skeleton: ")削除 -> "embedding_text"へ変換

Ground Truth構築（GT Creation）
├─ ground_truth_finder.py       → potential_gt.json
│  └─ LLMによる候補の自動抽出と審査
└─ leetcode_match_pgt.py        → ground_truth_v2.json
   └─ 100セットBJ-LC正解ペア完成

モデル評価（Model Benchmarking）
├─ evaluate_model.py            → OpenAI text-embedding-3-small
├─ evalute_model_ver2.py        → Jina-v3 (1024 dim)
└─ evalute_model_ver3.py        → BGE-M3 (ローカル)
   └─ Recall@K、MRR測定

---

この翻訳では、「データ前処理」から「モデル評価」までのプロセスと各ステップの詳細なタスクを日本語に翻訳しました。技術用語や特殊文字はそのまま維持し、日本語テクノロジーブログスタイルに合わせた表現を使用しています。

### なぜ100個だけなのか？

Ground Truthを100個作成したため、ランダムに100個ずつ抽出して実験を行いました。通常Golden Set（？）と呼ばれているものだと思います。これは人が直接マッピングする必要があるとのことでした。

##  임베딩 모델 비교 및 검증 (Embedding Model Benchmark)

最も精密な知識マッピングを行うために、3つの主要なエンベディングモデルを対象に性能検証（パイロットテスト）を行いました。

**比較モデル：Jina v3, OpenAI text-embedding-3-small, BGE-M3.**

### 評価指標 (Metrics)
* Recall@K: 上位 K 個の結果中で正解が含まれている割合。
* MRR（Mean Reciprocal Rank）: 正解が現れる順位の逆数の平均。

### ベンチマーク結果 (100-Set Golden Set基準)
| モデル | Recall@1 | Recall@5 | Recall@10 | MRR |
| ----- | ----- | ----- | ----- | ----- |
| OpenAI-v3 (small) | 0.22 | 0.50 | 0.65 | 0.354 |
| Jina-v3 (1024 dim) | 0.17 | 0.47 | 0.63 | 0.319 |
| BGE-M3 (Local) | 0.15 | 0.40 | 0.53 | 0.272 |

OpenAI-v3(small)モデルが最も優れた性能を示しました。

**Recall@10 :** 正解候補群を上位10位以内にうまく取り込むことができますが、 **Recall@1 :** 上位一位で正解を当てる割合は相対的に低い（22%）

### シルエットスコア
K-meansなどのクラスタリングアルゴリズムを使用してデータをグルーピングした際、グループが適切に分類されているかどうかを確認する方法の一つです。

これは[-1~1]の範囲の値を持ちます。私のプログラムでは、Baekjoon問題とLeetcode問題を正確にマッチさせることが目的であるため、$a(i) > b(i)$となるように、つまり、Baekjoon問題同士の距離($a$)が、Baekjoon問題とLeetcode問題間の距離($b$)よりも大きいことが望ましいです。そのため、負の値が出力される方が良いです。

$$
s(i) = \frac{b(i) - a(i)}{\max(a(i), b(i))}
$$

- $a(i)$ (クラスタ内距離): 自身が所属するクラスター内の他のデータとの平均距離です。
- $b(i)$ (クラスタ間距離): 所属していない最も近いクラスターとの平均距離です。

![silhouette](/images/ALGO_RAG/stage2_embedding_quality.png)

#### 1. シルエットスコア

RawからPreprocessedへ移行する際、最も大きな改善が見られました。一方でRefinedに進むとその傾向が逆転し、減少します。これは前処理の過程がエンベディング品質に最大の影響を与えたことを示しています。

#### 2. クラスタ内距離 (クラスター内部の距離)
クラスタ内のデータ同士がどれほど近いかを測定します。値が低いほど、同じカテゴリの問題がベクトル空間内で細かく集まり、密な配置ができていることを意味します。

Baekjoon（青色）はRawからRefinedへと進むにつれて距離が急激に減少しています。これは前処理を経てBaekjoonの問題が相互一貫性のあるベクトル表現を持つようになったことを示しています。一方、LeetcodeではPreprocessedでピークを打ち、その後下降する傾向を見せています。

#### 3. クラスタ間距離 (クラスタ間の距離)
異なるクラスタ（백준グループ vs 리트코드 그룹）間の距離を測定します。値が大きいほど、2つのグループは明確に区別されていることを示しています。

バックジョーンは段階が進むにつれて距離が逆に短くなり、これはバックジョーンデータがリトコードデータのベクトル領域に徐々に近づいていることを暗示しています。つまり、「意味的な距離」が2つのプラットフォーム間で縮まっていると解釈することができます。

#### 4. クラスタ品質比 (クラスタ内/外比)
(クラスタ内の距離 / クラスタ間の距離) の比率です。この値が低いほど、理想的な状態といえます。

両データセットはともにPreprocessed段階で最低点（最も良い品質）を記録します。その後Refined段階でわずかに上昇するのは、過度な精製によって一部の特徴的なデータが希釈された可能性があることを示しています。

#### 5. 距離分布 (ボックスプロット)
バックジョーンデータの距離分布を視覚化したものです。中央値（ボックス内の線）とデータの広がり具合を見ることができます。

クラスタ内：段階が進むにつれて、ボックスの高さは低くなり、位置は下に移動します。これはデータが極端な異常値（外れ値）を持たずに均等に密集していることを示しています。

クラスタ間：外部との距離分布も安定化し、他のデータセットと比較可能な範囲内に入っています。

#### 6. エンベディング品質ヒートマップ
すべての指標を統合して視覚化した表です。緑色に近いほど、その指標で優れたパフォーマンスを示しています。

**BJ Prep（バックジョーン前処理）**とLC Prep（リトコード前処理）列では緑色の割合が最も高くなっています。

結果として、現在のデータセットでは「Preprocessed」状態でのエンベディングが検索効率が最も高いという統計的な結論を導くことができます。

## 結論と次のステップ

今回のインジェクションモデル比較およびデータ前処理分析により、以下の結論を得ました。
1. OpenAIのtext-embedding-3-smallモデルがJina v3よりも優れた性能を示しました。
2. データ前処理プロセスがエンベディング品質に大きな影響を与えることがわかりました。
3. Refinedステージでの過度な精査は、むしろパフォーマンスの低下を招く可能性があると確認されました。

まず、refined段階で論理的骨格を削除し、2つのデータセットを均等に揃えてから、もう一度ベンチマークを実施してみようと思います。
